# K416经验：图鉴开发稳定路径（浓缩版）

本文档用于沉淀 K416 已验证流程，作为后续其他枪皮（如 AUG、SCARH 等）的复用基线。

## 一、编码与映射基线

### 1) 品质/材质编码
- 品质：`U=优品`，`J=极品`
- 材质：`T=透光`，`G=贵金属`，`Q=其他`，`L=镭射`，`M=漆面`，`Z=木质`
- 复合材质：最多允许 2 位组合（如 `LZ=镭射+木质`、`GM=贵金属+漆面`）

### 2) 颜色编码（当前生效）
- `01` 白、`02` 红、`03` 黄、`04` 青、`05` 紫、`06` 棕
- `07` 黑、`08` 灰、`09` 橙、`10` 绿、`11` 蓝、`12` 粉
- 单色：第二色码固定 `00`（如 `1100` = 蓝色单色）
- 未知：`1111`

### 3) 命名主键
- 统一主键：`K4-<编码>-<流水号>`
- 图片命名：`K4-<编码>-<流水号>_[A|B|C|D].png`
- 目录名可为：
  - 常规：`UQxxxx` / `UGxxxx`（默认流水 `001`）
  - 扩展：`UQxxxx002` / `UGxxxx003`（表示 `002`/`003` 流水）

关键原则：**编码是唯一主键，名称/评价是展示字段，不参与文件名。**

## 二、四图采集与标准化

### 1) 图位定义
- `A`：市场缩略图（`Snipaste`）
- `B`：市场查看图（外观详情）
- `C`：室内持枪图
- `D`：室外持枪图

### 2) 裁剪规则（已实战确认）
- `A`：不裁剪
- `B`：左 `15%`、上 `10%`、右 `6%`、下 `18%`
- `C`：仅上裁 `2.2%`
- `D`：仅上裁 `2.2%`

说明：
- `C/D` 不做横向裁剪，保留左上角锚点信息（如武器名）。
- `B` 的目标是保留整枪主体并减少无效背景。

## 三、网页布局模式（K416 已验证）

## 1) 首页（总封面）
- 使用“枪皮总封面”做武器入口卡片。
- 卡片标题显示全称（例：`SCARH-电玩高手`）。
- 已完成武器可点击进入，未完成武器显示“建设中”。

## 2) 武器列表页（K416）
- 顶部：教程区（挑选注意事项 + 教程图）
- 中部：筛选条（品质、材质、颜色）+ 搜索框
- 下部：枪皮卡片网格（A 图 + 名称 + 编码 + 标签）

## 3) 详情页（点入单皮肤）
- 主图：B（大）
- 辅图：C、D（下方双列）
- 信息：武器、品质、材质、配色、评分/简评（可选）
- 讨论看板：预留占位

## 四、详情页三图查看交互（重点）

- B/C/D 均支持点击大图查看（Lightbox）
- 关闭方式：
  - 右上角关闭按钮
  - 点击遮罩空白区域
  - `Esc` 键

该交互已验证可解决下方 C/D 图“看不清细节”的问题。

## 五、卡片交互与筛选体验

- 卡片悬停：原位膨胀（可轻微压住邻近卡片），增强可读性
- 筛选逻辑：
  - 品质/材质/颜色可组合筛选
  - 与搜索关键词联动（编码、名称、标签）

## 六、数据文件职责（开发时不要混用）

- `K416/k416_index_step1.csv`
  - 索引主表（id、路径、状态）
- `site/data.js`
  - 前端列表数据（由索引生成）
- `site/meta.js`
  - 展示层字段（name、rating、comment）
- `site/covers.js`
  - 首页武器封面入口

## 七、扩展到其他枪皮的固定流程

1. 采集该武器四图（A/B/C/D）
2. 按编码规则重命名并裁剪
3. 更新索引表并生成前端数据
4. 在总封面加入该武器入口卡
5. 在 `meta.js` 添加对应 id 的 name/rating/comment
6. 页面验收（列表、详情、放大、筛选、搜索）

## 八、每次改动后的验收清单

- 是否存在非标准目录名（应为 0）
- 是否存在 `incomplete` 条目（应为 0）
- 首页封面是否可进入对应已开发武器
- 列表筛选是否与搜索联动
- 详情 B/C/D 是否都可点击放大
- 名称修改是否仅改 `meta.js`（不改主键与图片名）

## 九、当前阶段进度总结（2026-05-30）

本节记录从 K416 单枪 MVP 扩展到多武器图鉴后的当前状态，供后续重新构思结构前参考。

### 1) 当前网站形态
- 项目仍是纯静态网站：`site/index.html` + `site/styles.css` + `site/app.js` + `site/data.js` + `site/meta.js` + `site/covers.js`。
- 前端不依赖后端接口，不使用数据库，不使用构建工具；所有数据直接由 JS 文件挂到 `window` 上。
- 当前适合通过 OSS / 静态网站托管发布。已整理出干净上传包：`skin-wiki/`。
- OSS 上传时建议保持目录结构：`site/` 与各武器资源目录同级，否则 `../武器目录/...` 相对路径会失效。

### 2) 已接入或已整理的武器
- 已完成网站接入并可在图鉴中浏览：
  - `K416`
  - `AUG`
  - `QBZ95`
  - `SCARH`
  - `Vector`
  - `KC17`
  - `M4A1`
  - `腾龙`
- `ASVAL` 曾经执行过录入与命名，但随后按“回到录入之前”的要求回退；当前不应视为已接入武器。
- `M250`、`M7`、`MP7` 等入口可保留在 `covers.js` 中作为建设中条目，但未完成数据接入时保持 `enabled: false`。

### 3) 数据层约定
- `site/data.js` 是网站实际渲染的主数据源，包含每个皮肤的：
  - `id`
  - `folderCode`
  - `normalizedCode`
  - `weapon`
  - `serial`
  - `imageA/imageB/imageC/imageD`
  - `status`
  - 可选 `template/qualityLabel/materialLabel/colorLabel`
- `site/meta.js` 只维护展示层字段：
  - `name`
  - `rating`
  - `comment`
- 后续改名优先只改 `site/meta.js`，不要改 `id`、图片文件名或目录名。
- `site/covers.js` 负责首页武器入口、封面图和 `enabled` 状态。

### 4) 两类录入模式

#### K416/QBZ95 编码模式
- 主键形态：`武器-<编码>-<流水号>`。
- 适合有稳定材质/颜色编码的武器。
- `QBZ95` 已按类似 K416 的编码化方式整理，并进行过基于市场图的中文命名。

#### 模板名模式
- 主键形态：`武器-<模板名>-<流水号>`。
- 适合 `SCARH`、`Vector`、`AUG`、`KC17`、`M4A1` 这类目录名中中文后缀直接表示模板名的武器。
- 规则：品质/材质可作为标签，中文模板名进入 `template` 或默认展示名，不再强行拆颜色。
- 若后续出现“同模板名 + 同流水号 + 不同材质”的冲突，需要在主键层加入稳定消歧规则，避免详情页串数据。

### 4.2) 自动化重命名/裁剪共同规则

- 四图位仍统一为 `A/B/C/D`：
  - `A` 来自 `Snipaste_*.png`，直接复制，不裁剪。
  - `B/C/D` 来自 `Delta Force Screenshot*.png`。
- 裁剪参数目前在各脚本中基本一致：
  - `B`：左 `15%`、上 `10%`、右 `6%`、下 `18%`
  - `C/D`：仅上裁 `2.2%`
- `B/C/D` 分类策略存在两层：
  - 保底：按文件名/拍摄顺序。
  - 优先：用画面亮度与蓝色分量估算室外图，尽量把室外图判为 `D`。
- 因拍摄顺序并不总是可靠，新增武器时应保留 `bcdRule/sourceB/sourceC/sourceD` 到 mapping 表，方便人工复核。

### 4.3) 自动化命名与人工命名边界

- 自动化主键命名解决的是“唯一、稳定、可追踪”，不等同于最终展示名。
- 展示名分三类：
  - 模板直出：`SCARH`、`Vector`、`AUG`、`KC17`、`M4A1` 初始可直接使用模板名。
  - 规则池命名：`QBZ95` 使用 `DISPLAY_NAME_POOL` 先生成“意象+颜色”风格名称。
  - 看图命名：`腾龙` 等需要逐张看市场图后起名，避免只按编码直译。
- 后续如果用户要求“先入库、名字空着”，应写入 `data.js` 但让 `meta.js` 的 `name` 留空或暂不填，等看图后再补。
- 改展示名时优先只改 `site/meta.js`，不要改已经生成的主键与图片文件名。

### 4.4) 筛选逻辑差异

- 编码模式武器（如 `K416/QBZ95/腾龙`）：
  - `normalizedCode` 可解析出品质、材质、颜色。
  - 颜色筛选有意义。
- 模板名模式武器（如 `SCARH/Vector/AUG/KC17/M4A1`）：
  - 通常不做细分颜色，`colorLabel` 为 `NA` 或空。
  - 颜色筛选不应强求命中；主要靠品质、材质、搜索和模板名浏览。
- 复合材质武器：
  - 材质标签显示为 `镭射 + 木质` 一类组合。
  - 筛选时用“包含”逻辑，不用严格相等。

### 4.5) 脚本与武器策略对应关系

- `scripts/ingest_scarh_vector.ps1`
  - 专用于 `SCARH/Vector` 的早期纯模板名模式。
  - 去掉目录名前缀 `J`，主键不包含 `J`，但保留 `极品` 标签。
- `scripts/ingest_qbz95.py`
  - 专用于 `QBZ95` 编码化录入。
  - 会解析中文颜色、生成颜色码、重命名目录，并通过名称池生成展示名。
- `scripts/ingest_aug_template.py`
  - 专用于 `AUG` 的模板名录入。
  - 支持 1-2 位材质码，但不做颜色编码。
- `scripts/ingest_template_weapon.py`
  - 通用模板名脚本，已用于 `KC17/M4A1` 等。
  - 适合“前缀是品质/材质，后缀中文是模板名”的武器。
- `scripts/ingest_tenglong.py`
  - 专用于 `腾龙` 的编码化录入。
  - 支持颜色编码和特殊 `镭射贵金属` 复合材质场景。

### 8) 发布与上传包经验
- GitHub Pages 技术上可行，但本项目图片较多，Git 推送大包时容易受网络、代理和远端连接影响。
- 当前更适合使用阿里云 OSS 静态网站托管：
  - 上传 `skin-wiki/` 文件夹，或上传其内部内容到 Bucket 根目录。
  - 若上传外层 `skin-wiki/`，访问路径形如：`/skin-wiki/site/index.html`。
  - 若上传 `skin-wiki/` 内部内容到 Bucket 根目录，访问路径形如：`/site/index.html`。
- 干净上传包应只包含展示必需目录：
  - `site`
  - `K416`
  - `AUG`
  - `QBZ95`
  - `SCARH`
  - `Vector`
  - `KC17`
  - `M4A1`
  - `腾龙`
  - `枪皮总封面`
- 上传包中不应包含：
  - `.git`
  - `.cursor`
  - `scripts`
  - `浓缩SOP`
  - `_new_*.json`
  - 临时脚本或处理中间文件

### 9) 重要风险与教训
- 删除图片是高风险操作。标准化图、原始截图、索引文件在确认前不应轻易删除。
- 若用户表达“回退”，必须先确认回退粒度：
  - 仅下线网站入口
  - 移除 `data/meta/covers` 接入
  - 删除生成索引
  - 删除标准化图片
  - 删除原始截图
- 对于图片资产，建议以后采用“迁移/归档”优先，不直接删除；例如移动到 `_archive/` 或 `_raw_backup/`。
- 新武器录入前先明确：
  - 使用编码模式还是模板名模式
  - 是否需要颜色分类
  - 是否需要人工看图命名
  - 是否允许清理原始截图
  - 是否立即接入网站或只整理底层数据

### 10) 下一阶段重构可考虑的问题
- 是否继续让所有数据集中在 `site/data.js`，还是按武器拆成多个数据文件。
- 是否统一资源目录结构，例如 `assets/weapons/<weapon>/...`。
- 是否把 `上方教程` 改成数据配置，而不是写死在 `app.js`。
- 是否引入构建脚本生成上传包，避免手工复制。
- 是否保留中文目录名，或逐步改成 ASCII 目录名并在 `meta.js` 中保留中文展示名。
- 是否将“录入脚本/开发资料”和“发布包”彻底分层，固定生成 `skin-wiki/` 发布目录。
